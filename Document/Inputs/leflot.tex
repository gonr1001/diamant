\chapter{Le flot d'information}

Pour mieux comprendre la spécification d'un système il faut
d'abord comprendre le flot que l'information suit. Ainsi que tous
les transformations que les données passent pour devenir
information utile. Ce chapitre présente la cheminement et les
transformations des données.

C'est vrai que dans un système de production d'horaires
l'information doit suivre un certain chemin qui modifie les
données, à vrai dire, c'est le changement de l'information qui
permet à l'information d'acquérir une valeur ajoutée, mais ces
changements comportent des risques qu'il faut savoir controller.
Il faut pour tant, les systèmes d'information capables d'encadrer
les utilisateurs pour faire les changements de la manière
appropriée et contrôlée, pour éviter qu'ils introduisent des
nouveaux erreurs chaque fois qu'ils utilisent un logiciel.

Pour commencer, dans la figure\ref{flot1} nous montrons d'une
manière simplifiée le flot d'information. Nous montrons aussi les
entités participantes dans le processus, ces entités jouent chaque
une un rôle.

Les entités:
\begin{description}
\item[Serveur de l'Unité d'administration externe.]Dans
l'environnement définie à notre système, nous avons contact avec
une entité externe qui gère l'information académique de
l'Université. Elle utilise une base de données pour enregistrer
ses informations. Nous avons besoin d'interagir avec cette unité,
l'échange d'information est bidirectional, elle doit nous envoyer
un ou plusieurs fichiers dans format texte ou XML avec
l'information de départ pour la production des horaires, et nous
allons envoyer à la fin du processus les résultats de la
production des horaires.

Cette entité a certaines règles qu'il faut respecter pour pouvoir
recevoir et envoyer les données. Dans notre cas particulière, il
s'agit d'une entité très rigide où les changements sont difficiles
à faire. C'est pourquoi il nous faut un grand niveau d'adaptation
dans le côté de notre interface pour s'adapter à cette entité mais
aussi pour rester assez ouverte pour accueillir les données
d'entré d'autres entités.

\item[Base de données.]C'est ici que nous allons enregistrer les
données nécessaires pour la creation des horaires. Nous allons
utiliser certaines relationes comme une "mémoire tampon" dans
laquelle nous allons enregistrer les données bruts, c'est-à-dire,
tels qu'elles sont reçues du serveur de l'unité d'administration
externe. Ces données peuvent avoir des erreurs dans la
construction de chaque donnée, c'est pour cela qu'il faut
appliquer un filtrage. Ce filtrage consiste en valider les données
pour les laisser conformes à une structure de base de données
relationnelle normalisé antérieurement. Une fois que les données
sont filtrées, nous allons utiliser une deuxième partie de la base
de données pour stocker ces données filtrées et corrigées qui sont
prêts pour l'exploitation de notre application de creation
d'horaires. Chaque fois que des nouvelles données arrivent, il
faut appliquer le filtrage pour valider la coherence de
l'information.

\item[Diamant.]Il s'agit de notre logiciel de création d'horaires,
il est paramétrable et adaptable pour la création d'horaires de
cours, examens et autres types d'horaires. Il reçoit un ou
plusieurs fichiers d'entrée pour créer les horaires et il retourne
un ou plusieurs fichiers de données avec l'information de
l'horaire.
\end{description}

Une fois que nous avons compris le flot d'information initial,
nous allons le détailler. Pour montrer le chemin détaillée que
l'information traverse dans le système, nous utilisons la figure
\ref{flot2}. Dans cette figure nous avons marqué en couleur verte
les actions qui travaillent avec des données d'entrée ou sortie de
notre système, nous avons isolé le système à ce partie qui gère la
base de données. En rouge nous avons laisse les données qui ont
potentiellement des erreurs et qu'il faut vérifier. En bleu nous
avons les données qui sont déjà filtrées. Nous allons expliquer
chaque action~:


\begin{figure}[h]
\centering{\includegraphics[width=1\textwidth]{Images/flot2.eps}}
\caption{\emph{Le flot de l'information.}} \label{flot2}
\end{figure}

\begin{description}
\item[1. Génération externe.]Il y a certaines données qui sont
générées par le système externe que nous avons appelé "Unité
d'administration Externe". Cette entité est responsable
d'administrer l'information académique de l'Université et elle
génere certaines données que nous allons utiliser pour la création
des horaires. L'échange de données est faite à l'aide de certains
fichiers qui sont crées par un "job", qui est un processus
informatisé precise. Les fichiers peuvent venir en format texte ou
bien en format XML, c'est pourquoi notre système doit être capable
de maîtriser les deux formats. Nous allons analyser la structure
de ces fichiers plus tard.

\item[2. Charge de données.]Pour pouvoir modifier et mieux gérer
les données nous pensons que la meilleure outil est un système de
base de données, parce que ce système est conçue pour faciliter
l'accès et les modification à l'information. Dans une première
étape nous allons enregistrer les données bruts, c'est-à-dire,
tels qu'ils sont reçues dans quelques relations temporaires de la
base de données que nous allons appelées "tampon". Tous les
actions de filtrage et correction d'erreurs sont faites dans cette
partie de la base de données.

\item[3. Filtrage de données.]Dans l'étape précédente nous avons
chargé les données dans la base de données, mais nous ne
connaissons pas l'état de ces données. Par experience, nous savons
que la plupart des erreurs dans un logiciel sont attachées aux
erreurs dans la composition des données. Pour assurer la propreté
et la coherence des données il va falloir appliquer un processus
de filtrage, c'est-à-dire, nous prenons les données bruts et nous
faisons certains vérifications de l'intégrité et la coherence du
format. Tous les résultats du filtrage restent dans la partie
"tampon" de la base de données.

\item[4. Rapport d'erreurs.]Une fois que nous avons appliqué le
filtre des données, il faut créer un rapport d'erreurs. Ce
document nous montré tous les anomalies trouvées dans les données
reçues, il est une outil importante pour l'étape suivante.

\item[5. Correction d'erreurs.]Avec l'aide du rapport d'erreurs il
y a un utilisateur qui devra réaliser les changements nécessaires
pour corriger les erreurs. Une fois faites les corrections,
l'utilisateur peut re-faire les pas 3, 4 et 5, jusqu'au point
qu'il n'y aie plus d'erreurs, ou les erreurs soient minimes.

\item[6. Commit.]Une fois qu'il n'y a pas d'erreurs dans le
rapport d'erreurs, l'utilisateur peut executer la commande
"Commit", qui va transférer les données de la partie "Tampon" de
la base de données à la partie des "Données corrigées". Ces
données serviront à la création des horaires.

\item[7. Ajout et modification des données.]Si l'utilisateur
considère qu'il faut modifier certaines données avant de créer
l'horaire il doit utiliser une interface qui lui permettra cette
activité. Il peut ajouter certains données, modifier quelques
autres, et même effacer quelques données. Pour ces modifications
il y a certains règles à accomplir.

\item[8. Generation du fichier.]Avec les données dans la partie
des "données corrigées" de la base de données, l'utilisateur peut
créer un fichier en format XML avec toutes les données nécessaires
pour l'operation du logiciel \diamant{}, ce fichier tiens en
consideration tous les données ajoutées, ou modifies dans la base
de données.

\item[9. Utilisation par \diamant{}.]Avec les fichiers qu'il a
crée dans l'étape 8, l'utilisateur est capable de créer son
horaire. Des fois, l'utilisateur qui participe dans cette étape
est quelqu'un différente de l'utilisateur qui a fait la saisie des
données.

\item[10. Fichier horaire.]Une fois que le logiciel \diamant{} a
été exécuté et l'horaire crée, il faut générer un fichier sortie
avec les données de l'horaire. Ce fichier est crée en utilisant le
format XML.

\item[11. Charge de données.]Avec l'execution de \diamant{}, il y
a certains données qui sont modifies. Pour assurer la coherence de
l'information il faut les charger dans la base de données. Mais,
il faut faire attention, parce que ne sont pas toutes les données
qui ont été modifies. C'est pourquoi il faut appliquer un
processus d'importation selective, les données à charger sont
stockes dans la partie mémoire "Tampon2" de notre base de données.

\item[12. Filtrage de données.] Étant donnée que nous voulons
introduire des nouvelles données à la base de données, il faut
toujours vérifier la validité de ces données, c'est pourquoi nous
allons appliquer ici les filtres nécessaires pour assurer la
coherence des données.

\item[13. Commit.]S'il n'y a pas d'erreurs dans le filtrage des
données, nous pouvons appliquer "commit" à l'information et passer
l'information à la section des "Données Corrigées".

\item[14. Generation des fichiers.]Pour retourner l'information à
l'Unité d'Administration externe, il faut générer un ou plusieurs
fichiers en format texte ou XML contenant les modifications à
l'information de base ainsi que l'information concernant à
l'horaire qui a été crée. Dans cette étape il faut créer ces
fichiers.

\item[15. Envoi des fichiers.]Une fois que le ou les fichiers sont
crées, il faut les envoyer à l'Unité d'administration externe.
Avec cette activité le cycle de création d'un horaire est terminé.

\end{description}
