\section{Livrable 1}

Ce livrable sert à vérifier que les fichiers .sig sont corrects.
\dia{} peut bénéficier de ce livrable car si l'on utilise \dx{} pour
vérifier les fichiers à importer, on est sûr qu'il n'y a pas d'erreurs ou
des problèmes sur ces fichiers. \dia{} ne fait pas de vérification et
une erreur sur un fichier peut faire planter le logiciel ou pire
construire un horaire avec des erreurs.

\subsection{Caractéristiques externes}
Dans ce livrable les menus \pro{Fichier}, \pro{Preferences} et \pro{Aide} sont fonctionnels.

Dans le menu \pro{Fichier} les sous-menus \pro{Définir fichiers à importer}
et \pro{Importer automatiquement}
sont fonctionnels. Le sous-menu \pro{Définir fichiers à importer} permet d'afficher une boîte de dialogue afin
de choisir les fichiers à importer\footnote{Il s'agit de fichier à la \saphir{}}; ensuite une autre boîte de dialogue permet
d'indiquer le nom du fichier \pro{.dim} nécessaire à sauvegarder les nom des fichiers.
Le sous-menu \pro{Importer automatiquement} permet d'afficher une boîte de dialogue afin
de choisir le nom du fichier \pro{.dim} nécessaire pour faire l'importation.

Dans le cas de boîtes de dialogue, où plusieurs réponses sont possibles,
Il faut traiter tous les cas. Par exemple, buttons \pro{Ok} et
\pro{Annuler}.



Si pendant l'importation une erreur arrive une boîte de dialogue indique le problème
rencontré.
Si l'importation se passe sans erreur une boîte de dialogue indique le succès de
l'importation.

Dans le menu \pro{Preferences} le sous-menu \pro{Options L \& F}
est fonctionnel. Une boîte de dialogue est affichée et il est possible de
changer de Look \& Feel de toute l'application.

Dans le menu \pro{Aide} le sous-menu \pro{À propos de Diamant 1.5}
est fonctionnel. Une boîte de dialogue est affichée.

\section{Livrable 3}

Tout ce qui était présent dans les livrables precedents continuera
à fonctionner. Dans le cas de boîtes de dialogue, où plusieurs
réponses sont possibles, tous les cas seront traités. Ce livrable
est plus qu'un point de contrôle pour montrer que tout avance
bien. Il sera possible la construction d'un horaire manuellement
et son exportation au SIG. Il fait appel à une bonne partie de
l'optimisation, en particulier à l'initialisation de données et
calcul de conflits. En un mot presque tout sera fonctionnel mais
les détails de confort d'utilisation vont venir plus tard. L'idée
de ce livrable est de le donner aux utilisateurs afin qu'ils
puissent l'utiliser et avec leurs commentaires fixer des priorités
pour la suite du développement.


\subsection{Caractéristiques externes}

Dans \dx{} nous avons introduit la notion d'événement, qui
représente une entité à placer dans une période. C'est comme un
rendez-vous pris par plusieurs personnes pour assister à une
réunion. Un événement est défini par une activité, un  type, une
section (groupe) et une unité. Cette dernière  contient l'heure du
début, la durée, l'enseignant et le local.

Il ressemble au bloc de \dia{}, mais il est défini de manière plus
générale, donnant plus de flexibilité. Un événement n'est pas
spécifié directement par des données dans un fichier, il  est
déduit à partir des données contenues dans les fichiers.

Dans ce livrable les menus \pro{Fichier} et \pro{Affectation} sont
complets et fonctionnels, le menu \pro{Optimisation} est
partiellement fonctionnel.

Dans le menu \pro{Fichier} le sous-menu \pro{Exporter} est
fonctionnel, il sera donc possible de exporter un horaire et le
valider au SIG.

Dans le menu \pro{Affectation} les sous-menus \pro{Activités},
\pro{Événement}, \pro{Enseignant}, \pro{Groupes}, \pro{Locaux}
\pro{Définir ensembles} et \pro{Commencer horaire} sont tous
fonctionnels sauf \pro{Définir ensembles}.

Le sous-menu \pro{Activités} affiche une boîte de dialogue avec
deux listes d'activités : celles inclues et celle exclues. Une
activité est soit inclue avec tous ses événements associés soit
elle est exclue. Une activité qui est inclue, veut dire que tous
les événements associées à l'activité seront à l'horaire. Une
activité qui est exclue veut dire que pour cette activité aucun de
ces événements associes ne sera pas à l'horaire.

%%%%%%%%%%%

Le sous-menu \pro{Événements} affiche une boîte de dialogue avec
trois listes d'événements  figés, placés et non placées. Au départ
un événement figé est un bloc figé  au SIG. L'utilisateur peut
aussi le figer, une fois que la construction de l'horaire est
démarrée. Un événement est placé par l'optimisation de \dx{} ou
par l'utilisateur. Un événement non placé, veut dire que il le
sera lorsque l'optimisation sera exécuté. Un événement non placé
n'apparaît pas  à l'horaire.

Le sous-menu \pro{Groupe} affiche une boîte de dialogue permettant
de choisir les étudiants qui seront figés dans un groupe (En
interne le groupe est une section).

Le sous-menu \pro{Enseignant} permet de voir la disponibilité d'un
enseignant.

Le sous-menu \pro{Locaux} permet de voir la disponibilité d'un
local. Similaire à la disponibilité des enseignants. Par default
un local est disponible tout le temps.

La disponibilité des enseignants et des locaux doit être fonction
de la grille,

Le sous-menu \pro{Définir ensembles} permet de créer des ensembles
afin de montrer dans la grille horaire les éléments appartenant à
un ensemble.

Dans le menu \pro{Optimisation} les sous-menus \pro{Groupes de
même taille}, \pro{Groupes min conflits} seront fonctionnels.

Les actions nécessaires à l'optimisation :


\begin{itemize}
    \item Affectation initiale.
    Elle consiste à placer dans la grille horaire tous
    les événements figés au SIG. Elle doit faire dans l'ordre :
    \begin{enumerate}
         \item Construction de la liste d'événements à partir de
    la liste d'activités.
         \item Construction de la matrice de conflits d'étudiants.
         \item Construction de la matrice de groupes d'étudiants.
         \item Affectation de chaque événement dans la grille horaire.
            \item Recherche de conflits d'étudiants.
    \item Recherche de conflits de locaux.
    L'on doit rechercher les conflits de locaux entre 2 événements
    et les conflits de capacité de local d'un  événement.
     \item Recherche de conflits d'enseignants.
     L'on doit rechercher les conflits d'enseignants entre
     2 événements et les conflits de disponibilité d'un enseignant.

    \end{enumerate}




    \item Implementation de l'algorithme de balancement des groupes d'étudiants.
    \item Amélioration manuelle.
    Elle consiste à sélectionner un événement
    et à rechercher les potentiels conflits
    (enseignants, locaux, étudiants)
    générés par cet événements dans la grille horaire.

\end{itemize}
